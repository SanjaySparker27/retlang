# retlang Usage Guide

This document walks through every CLI subcommand and every public library call, with copy-pasteable examples. If anything here disagrees with the code, the code is right and this document is the bug; please file an issue at <https://github.com/SanjaySparker27/retlang/issues>.

## Installation recap

```bash
git clone https://github.com/SanjaySparker27/retlang && cd retlang && pip install -e .
```

After install, the `retlang` command is on your `PATH` and `import retlang` works in Python.

## CLI reference

All subcommands accept `--help` for an autogenerated argument list.

```bash
retlang --help
retlang encrypt --help
```

### `encrypt`

Encrypts a plaintext message with a passphrase and writes the ciphertext to stdout.

**Arguments**

| Flag                    | Required | Description                                                                                          |
|-------------------------|----------|------------------------------------------------------------------------------------------------------|
| `TEXT` (positional)     | yes*     | The plaintext to encrypt. Mutually exclusive with `--infile`.                                        |
| `--infile PATH`         | yes*     | Read plaintext from a file. UTF-8 is assumed.                                                        |
| `-p, --passphrase PHRASE` | yes    | The shared secret. Anyone with this phrase can decrypt.                                              |
| `-a, --alphabet NAME`   | no       | Output alphabet. See "Choosing an alphabet" below. Default: `base64`.                                |
| `-s, --strength LEVEL`  | no       | PBKDF2 strength preset: `fast`, `normal` (default), `strong`, `paranoid`.                            |
| `--iterations INT`      | no       | Custom PBKDF2 iteration count. Mutually exclusive with `-s/--strength`.                              |
| `--wordmap PATH`        | no       | Path to a JSON wordmap file; see "Wordmap format" below.                                             |
| `--salt HEX`            | no       | 16-byte hex salt. If omitted, a fresh random salt is generated and stored in the envelope.           |
| `-o, --outfile PATH`    | no       | Write ciphertext to this file instead of stdout.                                                     |

\* Exactly one of the positional `TEXT` or `--infile` must be supplied.

**Examples**

```bash
# Minimal: default base64 alphabet, normal strength, random salt
retlang encrypt -p hunter2 "hello"

# Emoji-food alphabet with a wordmap
retlang encrypt -p moonlight -a emoji-food --wordmap wordmap.json "attack at midnight"

# Read a file, write a file, use runes
retlang encrypt \
  --infile letter.txt \
  -p moonlight \
  -a runes \
  -o letter.rune

# Deterministic: supply your own salt so the ciphertext is reproducible in tests
retlang encrypt -p moonlight --salt deadbeefcafebabe00112233445566ff "hello"

# Bump strength for a sensitive draft
retlang encrypt -p moonlight -s strong "some careful words"

# Paranoid mode plus astro glyphs
retlang encrypt -p moonlight -a astro -s paranoid "a quiet secret"

# Advanced: custom PBKDF2 iteration count
retlang encrypt -p moonlight --iterations 750000 "calibrated to my laptop"
```

### `decrypt`

Reverses `encrypt`. The alphabet and iteration count are read out of the envelope, so you do not pass them again.

**Arguments**

| Flag                  | Required | Description                                                      |
|-----------------------|----------|------------------------------------------------------------------|
| `CIPHER` (positional) | yes*     | The ciphertext. Mutually exclusive with `--infile`.              |
| `--infile PATH`       | yes*     | Read ciphertext from a file.                                     |
| `-p, --passphrase PHRASE` | yes  | The shared secret used at encrypt time.                          |
| `--wordmap PATH`      | no       | The same wordmap used at encrypt time. Required if one was used. |
| `-o, --outfile PATH`  | no       | Write plaintext to this file instead of stdout.                  |

\* Exactly one of the positional `CIPHER` or `--infile` must be supplied.

**Examples**

```bash
# Decrypt a ciphertext passed as an argument
retlang decrypt -p moonlight --wordmap wordmap.json "🍕🍔🍣🍜..."

# Decrypt a file back to plaintext
retlang decrypt --infile letter.rune -p moonlight
```

If the HMAC tag does not verify (wrong passphrase, tampered ciphertext, wrong wordmap), `decrypt` exits with a non-zero status and prints `integrity check failed`. No partial plaintext is ever released.

### `genkey`

Emits a cryptographically random passphrase you can share out-of-band.

**Arguments**

| Flag             | Required | Description                                                |
|------------------|----------|------------------------------------------------------------|
| `--length N`     | no       | Number of characters. Default: 32.                         |
| `--alphabet SET` | no       | Character set: `alnum` (default), `hex`, `words`.          |

**Examples**

```bash
retlang genkey
# -> "f4c8e2...long random string..."

retlang genkey --length 6 --alphabet words
# -> "otter-violet-quiet-river-anchor-moss"
```

The `words` variant emits a diceware-style hyphenated phrase that is easier to read aloud over the phone.

## Choosing an alphabet

Every alphabet encodes the same 6-bit output stream in a different visual system. The security is identical across alphabets; the choice is about aesthetics, output length, and where you plan to paste the ciphertext.

| ID | Name            | Chars per 6-bit group | Good for                                                                 |
|----|-----------------|-----------------------|--------------------------------------------------------------------------|
| 0  | `base64`        | 1                     | Default. Shortest output. Survives every clipboard, terminal, and form.  |
| 1  | `letters`       | 2                     | ASCII-only, but visually uniform (blocks of letters). Output is 2x long. |
| 2  | `numbers`       | 2                     | Looks like a serial number or PIN dump. Output is 2x long.               |
| 3  | `symbols`       | 1                     | Cursed-keyboard vibe. Same length as base64.                             |
| 4  | `emoji-smiley`  | 1                     | Playful, expressive. Char count matches base64; UTF-8 bytes are larger.  |
| 5  | `emoji-animals` | 1                     | Same as smiley but with a different flavor.                              |
| 6  | `emoji-food`    | 1                     | Looks like a menu. Good for disguising in a food chat.                   |
| 7  | `emoji-nature` | 1                      | Plants, weather, landscapes — reads like a gardening journal.             |
| 8  | `geometric`    | 1                      | Clean, minimalist, renders the same everywhere.                           |
| 9  | `runes`        | 1                      | Elder Futhark. Looks like a saga. May not render on all platforms.        |
| 10 | `astro`        | 1                      | Zodiac and planetary glyphs. Feels esoteric; narrow platform support.     |

**Rules of thumb:**

- If the ciphertext needs to survive email, commit messages, or a strict form field, pick `base64` or `symbols`.
- If you want the output to look like a particular domain of discourse (a horoscope, a menu, a weather log), pick the matching emoji or astro profile.
- `letters` and `numbers` double the output length because they spend 2 output characters per 6 input bits. Use them when pure ASCII matters and you want something more readable than base64.
- Emoji profiles are visually short but UTF-8-long. A 100-character emoji string can be several kilobytes of bytes — fine for messaging apps, worth checking if you're writing to a size-limited field.

The legacy name `emoji` still works as an alias for `emoji-smiley`, so older scripts do not need to change.

## Choosing a strength

The `-s/--strength` flag on `encrypt` selects a PBKDF2-HMAC-SHA256 iteration count. More iterations = more CPU time to derive the key, which means an offline brute-force attacker has to spend proportionally more time per passphrase guess. The honest cost is that decrypt pays the same bill.

| Level      | Iterations  | Typical time | Pick it when...                                                          |
|------------|------------:|-------------:|--------------------------------------------------------------------------|
| `fast`     |     100,000 |       ~50 ms | You want low-latency scripts, puzzles, or one-off notes.                 |
| `normal`   |     200,000 |      ~100 ms | Default. Fine for diaries, journals, everyday obfuscation.               |
| `strong`   |     500,000 |      ~250 ms | You want meaningful brute-force resistance for a single shared secret.   |
| `paranoid` |   1,000,000 |      ~500 ms | Maximum resistance within the toy-crypto threat model. Decrypt is slow.  |

Timings are rough; they depend heavily on your CPU. Measure if you care.

**A real caveat:** bumping strength does not change the underlying threat model. `retlang` is still a stylized, toy-grade cipher. `paranoid` only raises the cost of offline passphrase guessing; it does not turn the Vigenere-style layer into AES-GCM. If your threat model actually needs that, see [SECURITY.md](SECURITY.md) for the honest recommendation (`age`, `gpg`, or the `cryptography` library).

**Advanced override:** if none of the presets suit you, use `--iterations INT` instead. This is mutually exclusive with `-s/--strength` — specify one or the other, not both.

```bash
# Calibrated to ~750 ms on a specific laptop
retlang encrypt -p moonlight --iterations 750000 "extra paranoid"
```

## Python library reference

Every CLI flag has a keyword-argument counterpart in the Python API.

### `retlang.encrypt`

```python
def encrypt(
    plaintext: str,
    passphrase: str,
    *,
    alphabet: str = "base64",
    wordmap: dict[str, str] | None = None,
    strength: str = "normal",
    iterations: int | None = None,
) -> str:
    ...
```

Returns a printable string in the chosen alphabet, including the stored salt, iteration count, and the appended HMAC tag. `strength` and `iterations` are mutually exclusive: if you pass `iterations=...`, it overrides the preset; if you leave `iterations=None`, the `strength` preset is used.

**Example**

```python
from retlang import encrypt

token = encrypt(
    "meet me at midnight",
    passphrase="moonlight",
    alphabet="runes",
    strength="strong",
)
print(token)
```

### `retlang.decrypt`

```python
def decrypt(
    ciphertext: str,
    passphrase: str,
    *,
    wordmap: dict[str, str] | None = None,
) -> str:
    ...
```

Returns the original plaintext. Raises `retlang.IntegrityError` on HMAC mismatch and `retlang.FormatError` on a malformed token. The iteration count and alphabet are read from the envelope, so `decrypt` does not take either.

**Example**

```python
from retlang import decrypt, IntegrityError

try:
    plain = decrypt(token, passphrase="moonlight")
except IntegrityError:
    print("wrong passphrase or tampered message")
```

### `retlang.genkey`

```python
def genkey(length: int = 32, alphabet: str = "alnum") -> str:
    ...
```

### `retlang.list_alphabets`

```python
def list_alphabets() -> list[tuple[str, str, str]]:
    """Return (name, description, preview) triples for every registered alphabet."""
```

### `retlang.register_alphabet`

You can extend the library with your own 64-symbol alphabet.

```python
from retlang import register_alphabet

register_alphabet(
    name="hearts",
    description="64 heart emoji",
    symbols=["❤️", "🧡", "💛", "💚", ...]  # exactly 64 unique symbols
)
```

After registration, the new alphabet is selectable via `alphabet="hearts"` and appears in `list-alphabets`.

## Wordmap format

A wordmap is a JSON file mapping lowercase whole words in the plaintext to substitute phrases. Words are matched case-insensitively but the substitution is inserted verbatim. Punctuation is preserved around the word.

```json
{
  "attack": "sunrise",
  "midnight": "blue-canyon",
  "headquarters": "the-garden"
}
```

**Encrypt with the wordmap:**

```bash
retlang encrypt -p moonlight --wordmap wordmap.json "attack at midnight near headquarters"
```

**What happens internally:**

1. The plaintext is preprocessed to `"sunrise at blue-canyon near the-garden"`.
2. The preprocessed string is encrypted normally.
3. On decrypt, the inverse mapping is applied to recover `"attack at midnight near headquarters"`.

The wordmap **must be supplied at decrypt time too**, and it must be the same file. If you lose the wordmap, you lose the ability to recover the original words; you will only get the substituted form back.

Wordmaps are **not cryptography**. They are there to make the plaintext look like a different domain of discourse ("sunrise" reads like a weather report, not a battle plan). An attacker who guesses your substitutions can invert them trivially; the real secrecy comes from the passphrase.

## End-to-end walkthrough

```bash
# 1. Generate a shareable passphrase
PASS=$(retlang genkey --length 6 --alphabet words)
echo "$PASS"
# otter-violet-quiet-river-anchor-moss

# 2. Write a wordmap
cat > wordmap.json <<'EOF'
{ "attack": "sunrise", "midnight": "blue-canyon" }
EOF

# 3. Encrypt (emoji-food profile, strong strength)
retlang encrypt \
  -p "$PASS" \
  -a emoji-food \
  -s strong \
  --wordmap wordmap.json \
  -o note.emoji \
  "attack at midnight"

# 4. Share note.emoji and $PASS with the recipient through separate channels.

# 5. Recipient decrypts
retlang decrypt \
  --infile note.emoji \
  -p "$PASS" \
  --wordmap wordmap.json
# attack at midnight
```

## Exit codes

| Code | Meaning                                       |
|------|-----------------------------------------------|
| 0    | Success                                       |
| 1    | Generic error (bad arguments, I/O failure)    |
| 2    | Integrity check failed on decrypt             |
| 3    | Unknown alphabet or malformed ciphertext      |

Exit codes are stable across minor versions; please rely on them in scripts.
